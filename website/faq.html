Frequently Asked Questions

Common questions about Indexarr, privacy, data handling, and how the network operates.

Privacy & Data

No. Indexarr does not log, store, or track IP addresses of users who search or browse the index. There is no access log, analytics, or telemetry built into the application.

The web server logs that Docker or your OS might produce are standard reverse proxy behaviour and are under your control — Indexarr itself writes nothing to disk about who connects.

The bootstrap node (bootstrap.indexarr.net) that helps new nodes discover peers also does not log IP addresses in the Indexarr application. Key point: Indexarr is self-hosted. Your searches, browsing, and downloads are processed entirely on your own hardware. The bootstrap node only facilitates peer discovery for sync — it never sees your search queries.

No telemetry, crash reporting, update checking, or usage analytics of any kind. Indexarr does not phone home for tracking purposes.

The outbound connections your node makes depend on which workers you enable:

  • Bootstrap node — if sync is enabled, your node contacts bootstrap.indexarr.net to discover other peers. This is a lightweight Indexarr relay that participates in the gossip protocol and helps nodes find each other. It exchanges only torrent metadata and peer lists — never search queries or user data
  • DHT crawling — standard BitTorrent DHT traffic (if the crawler worker is enabled)

Indexarr stores only torrent metadata — never actual file content. The database contains:

  • Torrent records: info hash, name, size, file list, seed/peer counts
  • Classification data: content type, title, year, resolution, codec (parsed from names)
  • Sync state: peer URLs and sync sequence numbers for gossip protocol
  • Your contributor identity: a locally-generated Ed25519 keypair (never leaves your node unless you choose to sync)

No user accounts, search history, download history, or personally identifiable information is stored.

For social features (comments, votes, nuke suggestions), Indexarr generates a one-way fingerprint from your connection metadata. This is a SHA-256 hash — it cannot be reversed to recover your IP address or any other identifying information.

The fingerprint is used solely to prevent duplicate votes and allow users to delete their own comments. It is not linked to any account or identity system.

Network & Sync

Indexarr uses a bidirectional gossip protocol with three loops:

  • Export: Your node periodically packages new torrent discoveries into signed, content-addressed delta files
  • Discovery: Your node finds peers via the bootstrap node, DHT, HTTP peer exchange (PEX), or optionally XMPP
  • Gossip: Your node picks a handful of random healthy peers each round, pulls their manifests and downloads any new deltas it hasn't seen, then pushes its own new deltas back to those peers

The push phase is what makes the protocol bidirectional — after pulling from a peer, your node offers any deltas that peer hasn't seen yet. This is especially important for nodes behind NAT or firewalls that can't accept inbound connections: they can still contribute data to the network by pushing to peers they've connected to. Pushed deltas are authenticated with Ed25519 signatures and checked against the ban list before acceptance.

New nodes start by contacting the bootstrap node (bootstrap.indexarr.net) to get an initial peer list. From there, peer exchange propagates and the network becomes fully decentralized — the bootstrap is just another peer. All deltas are cryptographically signed with your contributor identity. Content addressing (SHA-256) means identical data is never transferred twice.

The bootstrap node (bootstrap.indexarr.net) is a lightweight, public Indexarr instance running on a VPS that serves as the initial entry point for new nodes joining the sync network. It runs three services:

  • Indexarr (sync only) — Participates in the gossip protocol to relay torrent metadata deltas between nodes. It does not crawl the DHT or resolve metadata — it only relays what other nodes share. When nodes behind NAT push their deltas to the bootstrap, it caches them in memory so other peers can pull them without needing a direct connection to the original node
  • Caddy — Provides TLS termination with automatic Let's Encrypt certificates
  • Prosody (XMPP) — Optional MUC-based peer discovery channel where nodes publish their contributor ID and URL

The bootstrap node's primary role is peer discovery and relay. When your node first starts with sync enabled, it contacts the bootstrap to get an initial list of peers via peer exchange (PEX). It also acts as a relay for nodes that can't accept inbound connections — those nodes push their deltas to the bootstrap, which then serves them to other peers on their behalf. Once your node has discovered other peers directly, it gossips with the full mesh — the bootstrap is just another peer at that point.

What the bootstrap does NOT do: It does not store IP addresses in its database (only peer URLs and contributor IDs). It does not see your search queries. It does not crawl or index content. It does not track which nodes request which data. If the bootstrap goes offline, existing nodes continue syncing via peers they've already discovered.

A contributor ID is a short identifier (format: TN-xxxxxxxx) derived from an Ed25519 public key generated on your node at first boot. It identifies your node in the sync network so other nodes can verify the authenticity of your shared data.

The private key never leaves your machine. A recovery key (base32-encoded) is shown once during onboarding so you can restore your identity if you reinstall.

Yes. The epoch system includes a ban list of contributor IDs. When a contributor is banned (e.g., for distributing harmful metadata), all nodes in the network will reject deltas signed by that identity.

Bans are enforced at the contributor identity level — not by IP address. This means bans are robust against IP changes while also avoiding any need to track network addresses.

Sync is entirely optional. By default, Indexarr runs with http_server, dht_crawler, and resolver workers. Your node will crawl the DHT independently and build its own index without connecting to any Indexarr infrastructure.

To enable sync, set INDEXARR_SYNC_ENABLED=true and add sync to your workers list. Your node will then contact the bootstrap node to discover peers and begin exchanging torrent metadata via the gossip protocol. You can disable sync at any time without losing data.

DHT & Crawling

The DHT crawler passively observes the public BitTorrent Distributed Hash Table — the same network used by every BitTorrent client. It discovers info hashes (unique torrent identifiers) being announced by peers worldwide.

Once an info hash is discovered, the resolver worker fetches the torrent's metadata (name, file list, size) using BEP 9 (metadata extension). This metadata is then parsed, classified, and stored in your local database.

The crawler does not download any actual file content — only metadata.

The DHT crawler participates in the BitTorrent DHT as a standard node, which means your node's IP is visible to other DHT participants — the same as running any BitTorrent client. This is inherent to how DHT works.

If this is a concern, you can:

  • Run the crawler behind a VPN (see the docker-compose.node-c.yml Gluetun example)
  • Disable the crawler entirely and rely on sync from other nodes for your index
  • Use the web UI and API workers only (--workers http_server)

No. Indexarr is an indexer, not a torrent client. It only collects and stores metadata — torrent names, file lists, sizes, and info hashes. It never downloads, seeds, or transfers actual file content.

The only BitTorrent file transfers Indexarr performs are sync delta files between Indexarr nodes (small NDJSON files containing torrent metadata).

Uploads

You can add torrents to your index in two ways:

  • .torrent file upload — Upload a .torrent file via the web UI or API. Indexarr extracts the metadata (name, file list, size, info hash) and runs the content classification pipeline. The original .torrent file is not stored — only the extracted metadata is kept
  • Magnet link — Submit a magnet URI. The info hash is extracted and queued for the resolver to fetch metadata via BEP 9

In both cases, only metadata is indexed. Indexarr never downloads or stores actual file content.

No. Indexarr does not log who uploaded what. There are no user accounts, no upload history, and no IP logging. The uploaded torrent is tagged with source="uploaded" to distinguish it from DHT-discovered or synced content, but nothing links that record back to you.

If your node has a contributor identity and sync is enabled, the torrent will carry your node's contributor ID in sync deltas — this identifies your node, not you personally, and is the same identity attached to all torrents your node exports.

No. Indexarr is an indexer, not a tracker. These are fundamentally different things:

  • A tracker coordinates peers for a specific torrent — it tells downloaders which IP addresses are seeding a file. Indexarr does not do this
  • An indexer catalogues torrent metadata (names, sizes, categories) so you can search and discover content. This is what Indexarr does

Indexarr does not run an HTTP or UDP tracker. It cannot facilitate downloads on its own. All torrents in the index rely on external trackers or the BitTorrent DHT for peer coordination.

Important: When you upload a .torrent file, the torrent must reference one or more working trackers (or rely on DHT). Indexarr extracts the info hash and metadata for search purposes, but the actual peer-to-peer swarm depends entirely on the trackers embedded in the torrent or DHT availability.

Do not upload private tracker torrents to Indexarr. While Indexarr detects and stores the private flag from .torrent files, uploading private tracker content carries serious risks:

  • Account ban — Private trackers strictly prohibit sharing their .torrent files or metadata outside the tracker. Uploading one to Indexarr (especially with sync enabled) could result in your tracker account being banned
  • Metadata propagation — If sync is enabled, the torrent's metadata (name, info hash, file list) will be shared across the Indexarr network via the gossip protocol. You cannot recall it once exported
  • Passkey exposure — Some private trackers embed unique passkeys in announce URLs within .torrent files. While Indexarr does not extract or store tracker URLs from uploads, the safest approach is to never upload private tracker .torrent files at all
Rule of thumb: Only upload torrents that use public trackers or are intended for public distribution. If a torrent came from a private tracker, do not upload it to Indexarr.

Uploaded torrents are treated the same as any other torrent in your index. If sync is enabled, their metadata (info hash, name, file list, size, classification, tags) will be included in your next export delta and shared with peers via the gossip protocol.

This means other Indexarr nodes on the network will be able to search for and find the torrent's metadata. The actual .torrent file is never shared — only the extracted metadata. Other nodes' users would still need access to a working tracker or DHT to actually download the content.

Integration & Usage

Indexarr implements the full Torznab/Newznab API with 30 categories. To add it:

  • In Prowlarr: Add a new Indexer → Generic Torznab → URL: http://your-indexarr:8080/api/torznab
  • In Sonarr/Radarr (direct): Settings → Indexers → Add Torznab → same URL

If you've set an API key in Indexarr, provide it in the API Key field. Otherwise, leave it blank.

SQLite works well for small to medium deployments (under ~500K torrents). Set INDEXARR_DB_BACKEND=sqlite and you're done — no external database needed.

PostgreSQL 17 is recommended for production and larger indexes. It provides full-text search via tsvector, better concurrent write performance, and JSONB support for TMDB cache.

Yes. The project includes a ready-made docker-compose.node-c.yml that routes all traffic through Gluetun + WireGuard (Mullvad). This is the recommended setup if you want to run the DHT crawler with VPN protection.

You can also run just the web UI and sync workers without a VPN, and let a separate VPN-protected node handle crawling.

Workers are independent loops that run inside the Indexarr process:

  • http_server — Web UI + all APIs (always needed)
  • dht_crawler — Discovers info hashes from the DHT
  • resolver — Fetches metadata for discovered hashes
  • sync — Gossip protocol for sharing with other nodes

Typical setups:

  • Full node: --all (all workers)
  • Web + sync only: --workers http_server,sync
  • Web UI only: --workers http_server
